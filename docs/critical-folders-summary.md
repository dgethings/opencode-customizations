# Critical Folders Summary

## Overview

This document summarizes critical folders in the opencode-customizations project.

## Critical Directories

### skills/
**Purpose:** Contains Opencode skill packages
**Criticality:** ⭐⭐⭐ (Core functionality)

**Structure:**
```
skills/
└── youtube-obsidian/
    ├── scripts/           # Python implementation
    ├── test_data/         # Test fixtures
    └── SKILL.md          # Skill documentation
```

**Why Critical:**
- Contains all skill packages that extend Opencode AI capabilities
- Skills are the primary output of this repository
- Each skill is self-contained with tests and documentation
- Skills are discovered and loaded dynamically by Opencode AI

**Entry Points:**
- `skills/youtube-obsidian/scripts/get_youtube_data.py` - Main script

### _bmad/
**Purpose:** Internal BMAD framework for structured development
**Criticality:** ⭐⭐ (Development methodology)

**Structure:**
```
_bmad/
├── core/               # Core workflow engine
├── bmm/                # BMAD methodology workflows
└── bmb/                # BMAD builder module
```

**Why Critical:**
- Provides structured development methodology
- Used for project planning and documentation
- Not part of public API (internal framework)
- Generates planning artifacts in `_bmad-output/`

**Integration:**
- Used by agents to follow BMAD methodology
- Generates PRD, architecture, and epics

### _bmad-output/
**Purpose:** Generated artifacts from BMAD workflows
**Criticality:** ⭐⭐ (Generated outputs)

**Structure:**
```
_bmad-output/
├── planning-artifacts/   # Planning outputs
│   ├── prd.md
│   ├── architecture.md
│   └── epics.md
└── implementation-artifacts/  # Implementation outputs
```

**Why Critical:**
- Contains generated planning documents
- Reference for brownfield PRD workflows
- Tracks workflow status in `bmm-workflow-status.yaml`

### scripts/
**Purpose:** Root-level utility scripts
**Criticality:** ⭐ (Development utilities)

**Structure:**
```
scripts/
└── capture_test_data.py  # Test data capture for youtube-obsidian
```

**Why Critical:**
- Provides utility scripts for development
- Helper scripts for testing and data capture
- Not core functionality but aids development

### evals/
**Purpose:** Evaluation scripts for testing skills
**Criticality:** ⭐ (Quality assurance)

**Structure:**
```
evals/
├── eval_youtube_obsidian.py      # Main evaluation script
└── test_eval_youtube_obsidian.py  # Evaluation tests
```

**Why Critical:**
- Evaluates skill performance
- Validates skill accuracy
- Ensures quality before deployment

### docs/
**Purpose:** Generated project documentation
**Criticality:** ⭐⭐⭐ (AI retrieval source)

**Structure:**
```
docs/
├── project-structure.md
├── technology-stack.md
├── architecture-patterns.md
├── comprehensive-analysis.md
├── source-tree-analysis.md
└── critical-folders-summary.md
```

**Why Critical:**
- Primary AI retrieval source for brownfield PRD
- Complete project documentation
- Reference for understanding project structure
- Generated by document-project workflow

**Entry Point:** `index.md` (to be generated)

### .github/
**Purpose:** GitHub configuration and workflows
**Criticality:** ⭐⭐ (CI/CD automation)

**Structure:**
```
.github/
└── workflows/
    └── test.yml  # CI/CD workflow
```

**Why Critical:**
- Automated testing on every push/PR
- Ensures code quality (linting, formatting, coverage)
- Integration tests with real APIs
- Coverage reporting with 80% threshold

## Criticality Levels

- ⭐⭐⭐ **Critical** - Core functionality, essential for repository purpose
- ⭐⭐ **Important** - Significant impact on development or operations
- ⭐ **Useful** - Aids development but not essential

## Directory Interactions

```
skills/ (output)
    ↑
    │ (BMAD methodology)
    │
_bmad/ (internal) → _bmad-output/ (generated)
                              │
                              │ (AI retrieval source)
                              ↓
docs/ (documentation)

scripts/ (utilities) → skills/ (development support)
    ↑
    │
evals/ (evaluation) → skills/ (quality assurance)
    ↑
    │ (automation)
.github/workflows/ (CI/CD)
```

## Development Workflow

1. **Create Skill:** Add skill package to `skills/`
2. **Implement:** Write Python scripts in `skills/<skill>/scripts/`
3. **Test:** Write tests and run via `scripts/capture_test_data.py`
4. **Evaluate:** Run evaluation scripts in `evals/`
5. **Document:** Add `SKILL.md` to skill directory
6. **CI/CD:** Push to GitHub, `.github/workflows/test.yml` runs automated checks
7. **BMAD Planning:** Use BMAD framework to plan new features
8. **Generate Docs:** Run document-project workflow to update `docs/`
